Проект ONL.IS — система управления кофейней/общепитом/ритейлом.
Стек: MySQL, PHP 7.4, SSE, jQuery, HTML, CSS, Bootstrap.
Архитектура: мультитенантная — каждый клиент на своём поддомене, своя БД, мастер-база для управления всеми клиентами. Одна БД клиента рассчитана на до 100 точек этого клиента (не 100 разных клиентов).
Репозиторий: https://github.com/abox922-bot/onlis
Текущая задача: полная разработка с нуля. Связь со старой системой/БД не рассматривается ни в каком виде (включая синхронизацию через триггеры) — эта идея уже была отклонена.
Главная задача при создании интерфейсов — максимально возможный уровень UI и UX.

=======================================================================
Принципы БД
=======================================================================

utf8mb4 везде, TINYINT(1) для флагов, DECIMAL(10,2) для денег, created_at / created_by / updated_at / updated_by на каждой таблице (там где необходимо исходя из логики), полные имена без аббревиатур, файлы и изображения не хранятся в БД.
updated_at и updated_by — NULL по умолчанию (NULL означает «запись ни разу не редактировалась»). FK для created_by / updated_by ссылаются на users.id.
Таблица users + users_auth — разделение учётных данных и персональных данных.
Поле is_active TINYINT(1) NOT NULL DEFAULT 1 в таблице users — флаг доступа пользователя в систему.

Флаги: TINYINT(1) NOT NULL DEFAULT 0 (или DEFAULT 1). Паттерн NULL/1 не используется.
Исключение: поля где NULL несёт отдельный смысл (например updated_at — NULL означает «не редактировалось», actual в users — NULL означает «уволен»).

Типы FK (критично для избежания ошибки 1215):
users.id = INT(10) UNSIGNED — все FK на users (created_by, updated_by, user_id и аналоги) должны быть INT(10) UNSIGNED.
countries.id = INT(10) UNSIGNED — все FK на countries (country_id и аналоги) должны быть INT(10) UNSIGNED.
Несовпадение типов FK вызывает ошибку MySQL 1215 Cannot add foreign key constraint.

=======================================================================
Система прав доступа (RBAC)
=======================================================================

Модель: Permission (slug) → Role (набор slug) → User (роль + опц. точка).

Таблицы БД:

permissions — справочник разрешений (slug, title). Заполняется при деплое и через миграции при добавлении новых модулей. created_by отсутствует — системная запись.
roles — роли (name, is_system). is_system=1 защищает стандартные роли от удаления. created_by/updated_by nullable — системные роли создаются без пользователя.
roles_permissions — связь роль → разрешения. ON DELETE CASCADE по обеим FK.
users_roles — связь пользователь → роль. obj_id NULL = глобально (все точки), число = конкретная точка. ON DELETE RESTRICT для role_id — роль нельзя удалить пока назначена пользователям. created_by nullable.

Стандартные роли (is_system=1): Владелец, Администратор, Управляющий, Кассир, Бухгалтер.

Иерархия slug — строго линейная цепочка сверху вниз:
  module → module.child → module.child.grandchild

fncCan поднимается ВВЕРХ по иерархии: при проверке 'module.child.grandchild' проверяет наличие
'module.child' и 'module' в правах пользователя. Родительский slug покрывает всех потомков.
Дочерний slug назначается роли только если нужно ОГРАНИЧИТЬ доступ внутри модуля.
Дочерний slug НЕ покрывает родительский и НЕ покрывает соседние ветки — иерархия работает только вверх по одной линейной цепочке, не вбок и не вниз.

Пример правильной иерархии (модуль Organizations):
  organizations               — полные права (реквизиты + ОПФ + организации)
    └── organizations.manage  — редактирование организаций
          └── organizations.manage.view — только просмотр

Добавление прав нового модуля — рутина из двух шагов:
1. INSERT в permissions (новые slug) — через migration-файл
2. INSERT в roles_permissions (назначить slug нужным ролям)

Функции в fncs.php:

// Загружает flat array slug для пользователя. $obj_id опционален.
function fncLoadPermissions(int $user_id, ?int $obj_id = null): array

// Проверяет право с учётом иерархии slug.
function fncCan(array $perms, string $slug): bool {
    if (in_array($slug, $perms)) return true;
    $parts = explode('.', $slug);
    for ($i = count($parts) - 1; $i > 0; $i--) {
        $parent = implode('.', array_slice($parts, 0, $i));
        if (in_array($parent, $perms)) return true;
    }
    return false;
}

Интеграция в app_X.php — после $user_id:

$user_id = (int)($ses_check['user'] ?? 0);
$perms   = $ses_check['rules'] ?? [];

Проверка в каждом case до выполнения логики:

if (!fncCan($perms, 'geography.edit')) {
    echo json_encode(['sccss' => false, 'msg' => 'Нет доступа']);
    exit;
}

Два уровня прав в модуле (пример Geography):

geography — полные права во всех файлах модуля
geography.edit — создание и редактирование (new_*, upd_* actions)
geography.delete — удаление (del_* actions) — добавляется когда реализуется UI удаления

in_cntrl проверяет is_active до загрузки прав. Заблокированный пользователь получает sccss:false и вылетает из системы немедленно.

Фронт (js/index.js):

Чеклист для нового модуля:
1. INSERT в permissions — новые slug (migration-файл)
2. INSERT в roles_permissions — назначить slug нужным ролям
3. Одна строка в menu_map.php (slug + module)
4. $perms = $ses_check['rules'] и fncCan в каждом case app_X.php
5. Проверка isset($result['sccss']) в каждом _list.php после send_request
6. canDo в JS: скрытие #btnFastNew + обёртка click обработчика

=======================================================================
Архитектура загрузки модулей
=======================================================================

Единый main_module.php, реестр в modules_map.php. Пункты меню несут только data-module и data-ttl. Раздел по умолчанию — флаг 'default'.
fncRequireSession() обязателен в каждом PHP-фрагменте, загружаемом через .load(). HTTP-статус всегда 200 — jQuery .load() вставляет тело только при 2xx.

Навигация между разделами модуля

Разделы переключаются через табы. В main_module.php генерируются кнопки .module-tab с data-target. fncBookNav() в _main.js обрабатывает клики: переключает класс active, скроллит таб в зону видимости (scrollIntoView), скроллит страницу вверх через #divScrollArea, загружает раздел через .load().
При одном разделе в модуле табы не отображаются — раздел загружается сразу.

=======================================================================
Модальные окна и Offcanvas
=======================================================================

#mainModal — основная модалка (bootstrap.Modal, инстанс main_modal). Заголовок #mainModalLabel, тело #mainModalBody. Открывается через main_modal.show() сразу после установки спиннера, до .load().

#modalOffcanvas — выезжающая панель ВНУТРИ #mainModal (bootstrap.Offcanvas, инстанс modalOffcanvas). Заголовок #modalOffcanvasLabel, тело #modalOffcanvasBody. Используется для вложенных мини-форм поверх уже открытой карточки (например — добавление реквизита в набор ОПФ, не закрывая карточку ОПФ). Паттерн открытия идентичен главной модалке: установить спиннер в body, заголовок, modalOffcanvas.show(), затем .load(). Закрывается своим штатным крестиком (data-bs-dismiss="offcanvas") — отдельная кнопка закрытия в разметке не нужна. Скрывать через modalOffcanvas.hide() после успешного сохранения.

#myOffcanvas — боковое меню навигации (bootstrap.Offcanvas, инстанс myOffcanvas). Не используется в логике модулей напрямую.

=======================================================================
Tom Select — стандарт для полей с поиском по большому справочнику
=======================================================================

Tom Select используется вместо кастомных dropdown-решений. Подключается через CDN в index.php. Инициализируется на <select> элементе.
В toolbar фильтр страны всегда через Tom Select (не обычный select).

let picker = new TomSelect("#slctElement", {
    placeholder: "Выберите...",
    allowEmptyOption: false,
    maxOptions: null,
    wrapperClass: "ts-wrapper toolbar-filter",
    plugins: ["clear_button"],
    render: {
        item: function(data, escape) {
            return `<div class="item">${escape(data.text)}</div>`;
        }
    },
    onChange: function(value) {
        if (value) {
            $("#divEmptyHint").addClass("d-none");
            $("#btnFastNew").prop("disabled", false);
            listLoadFunction(value);
            picker.blur();
        } else {
            $("#divChptContent").addClass("d-none").html("");
            $("#divEmptyHint").removeClass("d-none");
            $("#btnFastNew").prop("disabled", true);
        }
    }
});

При динамической загрузке через .load() объявлять через window.picker чтобы избежать ошибки redeclaration:

if (window.picker) window.picker.destroy();

=======================================================================
Стандарт форм — разметка
=======================================================================

Два типа форм: formNew (создание), formInfo (редактирование).
Контекстная информация (страна, регион, город) — через .form-context в отдельном col-12 перед полями, не через disabled-поля:

<div class="col-12">
    <div class="form-context">
        Россия <span style="opacity:0.35;margin:0 6px">›</span> Свердловская область
    </div>
</div>

Управляемые поля (участвуют в fncParamsCrt):

class="form-inp" — маркер для fncParamsCrt (обязателен)
class="form-in" — стили (обязателен отдельно)
data-name="..." — ключ параметра на сервер
data-type="..." — тип валидации
data-required="1" — если поле обязательное
data-length="N" — если значение должно быть ровно N символов
data-phone-code="..." — для типа phone, код страны для обрезки
data-phone-mask="..." — для типа phone, маска для подсчёта ожидаемой длины

Пример:

<input type="text"
    class="form-in form-inp"
    data-name="reg-code"
    data-type="digits_only"
    data-required="1"
    data-length="2">

Классы: form-in для инпутов и селектов, my-input-label для label. Не использовать form-control, form-select, form-label.
Первая опция select всегда value="0" с текстом-подсказкой.
placeholder используется там где нужна подсказка по формату (например маска телефона, часовой пояс).
Кнопка сохранения — всегда:

<button type="submit" class="btn-action-main" id="btnSave">
    <span id="btnSaveText">Сохранить</span>
    <div class="spinner-border spinner-border-sm d-none" id="divSaveLoading"></div>
</button>

Layout формы определяется контентом и контекстом — анализируется количество полей, их смысловые связи, приоритет визуального веса, поведение на мобильной версии. Bootstrap-сетка — инструмент, а не шаблон.

Условное отображение/доступность поля в зависимости от выбора в другом поле (например select меняет видимость блока) — решается на стороне PHP при первичном рендере (через классы d-none / disabled на основе данных из БД), чтобы избежать визуального мигания при .load(). JS-обработчик (change) добавляется отдельно только для живой реакции при ручном изменении поля пользователем уже после загрузки формы — без автоматического вызова при открытии (без trigger("change") на старте), если не оговорено иное.

=======================================================================
Стандарт форм — data-type и валидация
=======================================================================

Правило валидации: required добавляет только проверку на пустоту. Правила формата применяются всегда когда в поле есть хоть что-то. Если поле не required и пустое — ошибки нет, передаётся пустое значение.

text — убирает HTML-теги. required + пустое → ошибка.
digits_only — только цифры. required + пустое → ошибка.
digits_double — цифры и точка (запятая → точка). required + пустое → ошибка.
without_letters — убирает все буквы (рус/лат). required + пустое → ошибка.
email — убирает кириллицу и кавычки. Длина ≥ 5, есть @ и точка — всегда когда не пустое.
eng_text — только a-z0-9. required + пустое → ошибка.
select — значение ≠ 0 (всегда).
check — возвращает 1 или 0.
phone — только цифры, обрезает код страны. Длина совпадает с маской — всегда когда не пустое.

При ошибке валидации поле получает класс is-invalid (стиль в style.css). Поле подсвечивается до следующей попытки отправки.

Обработчик submit — обязательный паттерн:

$("#formNew").submit(function(e) {
    e.preventDefault();
    e.stopImmediatePropagation();
    let params_arr = [];
    params_arr.push({name: "key", value: value});
    let crt_arr = fncParamsCrt(".form-inp", params_arr);
    if (crt_arr["all_good"]) {
        $("#btnSave").prop("disabled", true);
        $("#btnSaveText, #divSaveLoading").toggleClass("d-none");
        fncMyAjax("action_name", "module", crt_arr["params"], 1)
            .done(function(data) {
                if (data.sccss) {
                    main_modal.hide();
                } else {
                    fncBtnReset();
                }
            })
            .fail(function() {
                fncBtnReset();
            })
            .always(function() {
                listLoadFunction();
            });
    }
});

e.preventDefault() + e.stopImmediatePropagation() — всегда оба. Критично для форм внутри модалок, загружаемых через .load() — без этого на мобильной версии и при повторном открытии обработчики накапливаются.

Паттерн для операций обновления (upd_*), return_data: 0 — тело ответа не возвращается:

fncMyAjax("upd_item", "module", crt_arr["params"], 0)
    .done(function() {
        main_modal.hide();
    })
    .fail(function() {
        fncBtnReset();
    })
    .always(function() {
        listLoadFunction();
    });

return_data: 1 — JS ожидает данные в ответе (нужен id или sccss). 0 — ответ не нужен, список перезагружается через .always().

Стандарт app_X.php (API модуля)
=======================================================================

<?php
require_once('./includes/fncs.php');
require_once('./includes/request.php');

header('Content-Type: application/json');

$cookie = $_POST['_onlis_id'] ?? '';
$token  = $_POST['x_token']   ?? '';

if (!$cookie || !$token) { echo json_encode(['sccss' => false]); exit; }

$ses_check = send_request([
    '_onlis_id' => $cookie,
    'x_token'   => $token,
    'action'    => 'in_cntrl'
], 'main');

if (!$ses_check || empty($ses_check['sccss'])) {
    echo json_encode(['sccss' => false]);
    exit;
}

$user_id = (int)($ses_check['user'] ?? 0);
$perms   = $ses_check['rules'] ?? [];
$action  = $_POST['action'] ?? '';
$params  = isset($_POST['params']) ? json_decode($_POST['params'], true) : [];
if (!is_array($params)) $params = [];

$result = [];

switch ($action) {
    case 'action_name':
        if (!fncCan($perms, 'module.action')) {
            echo json_encode(['sccss' => false, 'msg' => 'Нет доступа']);
            exit;
        }
        // логика
        break;
    default:
        echo json_encode(['sccss' => false, 'msg' => 'Неизвестное действие']);
        exit;
}

echo json_encode($result);

HTTP 200 всегда. Content-Type: application/json. Валидация через fncValFind. htmlspecialchars при выводе. created_by / updated_by берутся из $ses_check['user']. $perms загружается из $ses_check['rules'] — не отдельным запросом к БД.

Любой параметр, отправленный через params (массив, упакованный через fncParamsCrt и отправленный через fncMyAjax), читается на сервере через fncValFind('имя-параметра', $params) — НЕ напрямую через $_POST['имя']. $_POST используется только для параметров, отправленных напрямую вторым аргументом в fncMyAjax/.load() (например country_id, organization_type_id, id при простой загрузке через .load()), не через params_arr/fncParamsCrt.

=======================================================================
Подход к работе с кодом
=======================================================================

При правках существующего кода Claude показывает только изменяемый фрагмент (было/стало или новый кусок целиком, если добавляется новый блок) — НЕ файл целиком, если не запрошено явно. Алексей вносит изменения в код вручную и держит в голове контекст всего файла.

Перед любой вёрсткой (форма, список, toolbar, любой новый UI-элемент) — обязательный шаг анализа UI/UX: группировка полей по смыслу, визуальные отступы между смысловыми блоками, понятность подписей и подсказок, логические связи между полями (например — поясняющий текст под чекбоксом, если связанное значение задаётся в другом месте интерфейса). Это не опциональный шаг, он выполняется каждый раз перед написанием разметки, а не только когда явно попросили.
